Key outcomes from the session:

  • Indexing succeeded when we mounted the real repo folder directly into the container (not via a Windows symlink).
  • The indexed project name we ended up with was repo (because we mounted the repo at /repo).

Step 4.3 — Build the Docker image

The Dockerfile used in this session lives at C:\mcp-config\Dockerfile.

Step 4.3.1 — Dockerfile contents (C:\mcp-config\Dockerfile)

This is the exact Dockerfile content used in this session.

# codebase-memory-mcp — pre-built binary (pure C, not Python/Node)
# Uses stdio (stdin/stdout JSON-RPC) for MCP communication
FROM ubuntu:24.04

ARG CBM_VERSION=0.6.0

# Download the pre-built binary + install git for change detection
RUN apt-get update && apt-get install -y --no-install-recommends \
      curl ca-certificates git \
    && curl -fsSL "https://github.com/DeusData/codebase-memory-mcp/releases/download/v${CBM_VERSION}/codebase-memory-mcp-linux-amd64.tar.gz" \
      -o /tmp/cbm.tar.gz \
    && tar xzf /tmp/cbm.tar.gz -C /tmp \
    && mv /tmp/codebase-memory-mcp /usr/local/bin/codebase-memory-mcp \
    && chmod +x /usr/local/bin/codebase-memory-mcp \
    && rm -rf /tmp/* /var/lib/apt/lists/*

# Non-root user for safety
RUN useradd -m -d /home/cbm cbm \
    && mkdir -p /home/cbm/.cache/codebase-memory-mcp \
    && chown -R cbm:cbm /home/cbm/.cache
USER cbm

# Repos mounted here (via -v C:\mcp-workspace:/repos)
WORKDIR /repos

# Persist indexed graph data across restarts
VOLUME ["/home/cbm/.cache/codebase-memory-mcp"]

ENTRYPOINT ["codebase-memory-mcp"]

5) Docker image + container management

5.0 Create the persistent cache volume (recommended)

This volume persists the indexed graph so you don’t lose it between runs.

5.1 Verify the CLI works

docker run --rm -i mcp-memory --help

5.2 Inspect a running container

We inspected the running mcp-memory container and saw that it did not expose ports and had a different mount than planned. These were the exact commands used: 

docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Status}}\t{{.Ports}}"

docker port mcp-memory

docker inspect mcp-memory --format "Image={{.Config.Image}} Cmd={{json .Config.Cmd}} ExposedPorts={{json .Config.ExposedPorts}} PortBindings={{json .HostConfig.PortBindings}} Binds={{json .HostConfig.Binds}}"

5.3 Important note about ports

codebase-memory-mcp is primarily an MCP server over stdio (stdin/stdout JSON-RPC). It can optionally expose an HTTP UI with --ui=true, but you still need to publish ports with -p when using Docker.

5.4 Optional: enable the HTTP UI

If you want the visualization UI, you need the UI build of codebase-memory-mcp (the plain MCP server build may not start an HTTP listener).

Important Windows/Docker detail:

The UI server binds to 127.0.0.1 inside the container. Docker port publishing can’t reach a loopback-only listener directly, so you must proxy it to 0.0.0.0 (example below).

5.4.1 Start the UI container (recommended)

This runs an Ubuntu container that downloads the UI-enabled binary codebase-memory-mcp-ui-linux-amd64, starts the UI on 127.0.0.1:9749, and uses socat to expose it on 0.0.0.0:9750. Host port 9749 is mapped to container port 9750. 

docker rm -f cbm-ui 2>$null

docker run -d -i -t --name cbm-ui ^
  -p 9749:9750 ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  ubuntu:24.04 bash -lc "set -e; \
    apt-get update >/dev/null; \
    apt-get install -y --no-install-recommends curl ca-certificates socat >/dev/null; \
    useradd -m -d /home/cbm cbm || true; \
    chown -R cbm:cbm /home/cbm; \
    nohup socat TCP-LISTEN:9750,fork,reuseaddr,bind=0.0.0.0 TCP:127.0.0.1:9749 >/tmp/socat.log 2>&1 & \
    su - cbm -c 'curl -fsSL https://github.com/DeusData/codebase-memory-mcp/releases/download/v0.6.0/codebase-memory-mcp-ui-linux-amd64.tar.gz -o /tmp/cbm-ui.tar.gz && tar xzf /tmp/cbm-ui.tar.gz -C /tmp && /tmp/codebase-memory-mcp --ui=true --port=9749'"

5.4.2 Open the UI

http://localhost:9749/

5.4.3 Stop / remove

docker stop cbm-ui
docker rm cbm-ui

5.4.4 If you see ERR_EMPTY_RESPONSE

  • Prefer IPv4 explicitly: http://127.0.0.1:9749/
  • Confirm the container is running: docker ps --filter name=cbm-ui
  • Check logs: docker logs --tail 200 cbm-ui

6) Indexing (the important part)

6.1 The symlink pitfall (why indexing initially failed)

We tried indexing via the symlink under C:\mcp-workspace, mounted into Docker at /repos. It failed. The indexer output showed:

  • pipeline.discover files=0
  • status":"error"

We confirmed the underlying issue by checking the mount from an Ubuntu container: 

docker run --rm -v C:\mcp-workspace:/repos ubuntu:24.04 bash -lc "ls -la /repos; echo '---'; ls -la /repos/intuneassignments-website | head"

Inside the Linux container, the symlink appeared as:

/repos/intuneassignments-website -> /mnt/host/c/GitHub/intuneassignments-website

That /mnt/host/... path does not exist inside the container, so the repo looked empty.

6.2 The working solution

Mount the real Windows repo folder directly into the container and index from that mount: 

docker run --rm ^
  -v C:\GitHub\intuneassignments-website:/repo ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  mcp-memory cli index_repository '{"repo_path":"/repo","mode":"moderate"}'

Then verify the project is registered in the persistent cache:

docker run --rm ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  mcp-memory cli list_projects '{}'

What we saw:

  • projects included repo with root_path = /repo
  • Index stats: nodes=756, edges=1396

6.3 Checking index status

Once indexed, you can query status for the project name (repo) using the CLI tool: 

docker run --rm ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  mcp-memory cli index_status '{"project":"repo"}'

If you remount the repo at a different container path, you may end up with a different project name. Keep the mount path stable to avoid multiple projects for the same repo.

6.4 Indexing multiple repos (C:\GitHub\repos)

If you want to index multiple repositories stored under a single parent folder (example: C:\GitHub\repos), the cleanest approach is to mount that parent folder directly into the container and then index each subfolder.

Why this matters on Windows:

  • Avoid using Windows symlinks inside a bind mount (like C:\mcp-workspace) as the Linux container may not be able to follow the symlink target, leading to files=0 during discovery.
  • Prefer real directories under C:\GitHub\repos.

6.4.1 Recommended folder layout

C:\GitHub\repos\
  intuneassignments-website\
  repoA\
  repoB\
  ...

6.4.2 Index a single repo under the parent mount

docker run --rm ^
  -v C:\GitHub\repos:/repos ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  mcp-memory cli index_repository '{"repo_path":"/repos/intuneassignments-website","mode":"moderate"}'

In this pattern, the indexed project name will usually be the last path segment (example: intuneassignments-website) instead of the generic repo.

6.4.3 Index all repos in a loop (PowerShell)

This indexes each immediate subdirectory under C:\GitHub\repos. 

$ReposRoot = "C:\GitHub\repos"

Get-ChildItem -Path $ReposRoot -Directory | ForEach-Object {
  $name = $_.Name
  Write-Host "Indexing $name..."

  docker run --rm ^
    -v ${ReposRoot}:/repos ^
    -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
    mcp-memory cli index_repository ("{`"repo_path`":`"/repos/$name`",`"mode`":`"moderate`"}")
}

7) Architecture and how it works

7.1 Components

  • VS Code Copilot Chat MCP Host: launches MCP servers configured in VS Code settings
  • codebase-memory-mcp (Dockerized): reads repo files, builds a graph, persists it to a cache directory
  • Persistent cache volume: Docker volume cbm-data mounted to /home/cbm/.cache/codebase-memory-mcp

7.2 Data flow

  1. Repo is mounted into container at a stable path (recommended: /workspace or /repo)
  2. Index tool walks files, extracts definitions/usages, and builds edges (calls/imports/usages/etc.)
  3. Graph is saved into the cache volume
  4. Future queries operate on the graph without re-reading everything unless re-indexed

7.3 Useful tools (high level)

Tool Use
index_repository Build/update graph
list_projects Show indexed projects
index_status Index status details
get_architecture High-level architecture summary
search_graph Find symbols by pattern / degree
trace_path Call graph tracing
query_graph Cypher queries over the graph

10) Troubleshooting checklist

A) "Indexing says indexed but list_projects is empty"

  • Ensure you are using the same cache location each time (same Docker volume mount).
  • If you index without mounting cbm-data, results won’t persist.

B) "discover files=0"

  • Confirm the mount path is correct by listing it from a throwaway container.
  • Avoid indexing through Windows symlinks inside a bind mount; mount the real repo path directly. 
docker run --rm -v C:\GitHub\intuneassignments-website:/repo ubuntu:24.04 bash -lc "ls -la /repo | head"

C) "Docker container has no ports"

  • That’s fine for MCP stdio mode; but if you expected a browser UI, you must enable UI and publish ports.

D) VS Code can’t see the server/tools

  • Verify chat.mcp.servers includes the server entry
  • Run Developer: Reload Window after editing settings
  • Make sure Docker is running and your user can run docker commands

E) Confirm your symlink is OK (Windows-side)

Get-Item 'C:\mcp-workspace\intuneassignments-website' | Format-List *

13) Backup & restore (containers + indexed data)

For sandbox validation and safe promotion to prod, the main thing you want to preserve is the indexed graph stored in the Docker volume cbm-data. Containers themselves are disposable.

13.1 What to back up

  • Required: Docker volume cbm-data (holds repo.db, workspace.db, config)
  • Optional: Docker image mcp-memory (so a sandbox can run without rebuilding)
  • Optional: VS Code user settings (your chat.mcp.servers configuration)

13.2 Backup the indexed data volume (cbm-data)

This creates a tar.gz of the volume contents to a folder on the host. 

# Choose a backup folder on the host
$BackupDir = "C:\\mcp-backups"
New-Item -ItemType Directory -Force -Path $BackupDir | Out-Null

# Create a timestamped backup file
$Stamp = Get-Date -Format "yyyyMMdd-HHmmss"
$BackupFile = "$BackupDir\\cbm-data-$Stamp.tgz"

docker run --rm ^
  -v cbm-data:/data ^
  -v ${BackupDir}:/backup ^
  ubuntu:24.04 bash -lc "tar czf /backup/$(basename $BackupFile) -C /data ."

13.3 Restore the indexed data volume (cbm-data)

This restores a previously created backup into a fresh cbm-data volume. 

$BackupDir = "C:\\mcp-backups"
$BackupFile = "$BackupDir\\cbm-data-YYYYMMDD-HHMMSS.tgz"  # replace

# WARNING: this deletes the existing indexed data volume
docker volume rm cbm-data
docker volume create cbm-data

docker run --rm ^
  -v cbm-data:/data ^
  -v ${BackupDir}:/backup ^
  ubuntu:24.04 bash -lc "tar xzf /backup/$(basename $BackupFile) -C /data"

13.4 Validate after restore

docker run --rm ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  mcp-memory cli list_projects '{}'

docker run --rm ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  mcp-memory cli index_status '{"project":"repo"}'

13.5 Backup/restore the mcp-memory image (optional)

Backing up the image is useful for air-gapped or slow sandbox environments so you don’t need to rebuild/pull anything. 

# Verify the image exists
docker images mcp-memory

# Backup (timestamped)
$BackupDir = "C:\\mcp-backups"
$Stamp = Get-Date -Format "yyyyMMdd-HHmmss"
$ImageBackup = "$BackupDir\\mcp-memory-$Stamp.tar"

docker save mcp-memory:latest -o $ImageBackup

# Restore
docker load -i $ImageBackup

13.6.1 Backup (export only the MCP servers config)

# Source settings
$Settings = "C:\\Users\\thiagobeier\\AppData\\Roaming\\Code\\User\\settings.json"

# Destination backup
$BackupDir = "C:\\mcp-backups"
New-Item -ItemType Directory -Force -Path $BackupDir | Out-Null
$Stamp = Get-Date -Format "yyyyMMdd-HHmmss"
$McpBackup = "$BackupDir\\vscode-chat.mcp.servers-$Stamp.json"

# Export only chat.mcp.servers
$json = Get-Content -Raw $Settings | ConvertFrom-Json
$json."chat.mcp.servers" | ConvertTo-Json -Depth 20 | Set-Content -Encoding UTF8 $McpBackup

Write-Host "Saved: $McpBackup"

13.6.2 Restore (merge into settings.json)

This merges the saved MCP servers map into your existing settings.json. It does not overwrite unrelated settings. 

$Settings = "C:\\Users\\thiagobeier\\AppData\\Roaming\\Code\\User\\settings.json"
$McpBackup = "C:\\mcp-backups\\vscode-chat.mcp.servers-YYYYMMDD-HHMMSS.json"  # replace

$settingsJson = Get-Content -Raw $Settings | ConvertFrom-Json
$mcpServers = Get-Content -Raw $McpBackup | ConvertFrom-Json

$settingsJson."chat.mcp.servers" = $mcpServers

$settingsJson | ConvertTo-Json -Depth 50 | Set-Content -Encoding UTF8 $Settings

Write-Host "Updated: $Settings"
Write-Host "Next: VS Code -> Developer: Reload Window"

13.7 Containers are disposable

You can always recreate containers from the image + volume. Examples: 

# Stop/remove UI container
docker stop cbm-ui
docker rm cbm-ui

# Re-run indexing on demand (recreates/updates graph data in cbm-data)
docker run --rm ^
  -v C:\\GitHub\\intuneassignments-website:/repo ^
  -v cbm-data:/home/cbm/.cache/codebase-memory-mcp ^
  mcp-memory cli index_repository '{"repo_path":"/repo","mode":"moderate"}'
